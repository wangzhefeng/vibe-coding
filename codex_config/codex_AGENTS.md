# AGENTS.md (System)

本文件是我电脑上的 Codex 系统级协作规范，适用于所有项目；仓库内更近层级的 `AGENTS.md` 可在不冲突时补充细节。

## 1) 协作目标
- 先确保需求被正确理解，再追求实现速度。
- 默认交付“可运行、可验证、可回滚”的改动。
- 优先减少返工：在编码前明确边界、风险、验收标准。

## 2) 接收任务时的默认动作
- 先复述目标：要做什么、做到什么算完成。
- 明确约束：语言/框架、运行环境、性能、安全、时间范围。
- 明确验证：测试命令、手动验证路径、预期输出。
- 对不明确项做“最小风险假设”并标注；高风险项先停下来确认。

## 3) 提示与执行最佳实践（遵循官方 Codex 建议）
- 任务描述使用四段式：`目标`、`上下文`、`约束`、`验收`。
- 复杂任务先给简短计划，再分步执行并持续同步进度。
- 优先小步提交：每次改动聚焦单一问题，方便审查与回退。
- 能自动验证就自动验证（测试/类型检查/构建），并汇报结果。
- 需要代码评审时，优先报告：缺陷、风险、回归点、测试缺口。

## 4) 代码探索与改动约定
- 搜索优先使用 `rg` / `rg --files`。
- 先读相关文件再改，避免盲改。
- 保持与项目现有风格一致；不引入无关重构。
- 注释简洁，重点解释“为什么”，避免解释显而易见的“做了什么”。
- 未经要求不做破坏性操作（如大规模删除、重置历史）。

## 5) 测试与质量门槛
- 能跑测试就跑最相关测试；若无法运行，明确说明原因与影响。
- 至少覆盖：正常路径、关键边界、已知高风险分支。
- 若变更影响接口/行为，补充最小必要测试或验证脚本。

## 6) Git 与交付
- 分支命名清晰，提交信息说明“改了什么 + 为什么”。
- 提交前检查：变更范围是否最小化、是否包含必要测试/文档更新、是否引入潜在机密或敏感信息。
- 输出结果时附：关键文件、关键命令、验证结论、残留风险。

## 7) 安全与边界
- 不输出、不硬编码密钥或凭证。
- 涉及生产、数据删除、不可逆操作时，先明确风险并征得确认。
- 对高风险领域（安全、财务、法律、医疗）默认提高校验强度。

## 8) 工具使用偏好
- 可并行的读操作尽量并行，提高反馈速度。
- 有现成脚本优先复用，不重复造轮子。
- 需要外部信息时优先官方文档/一手资料，并标注来源。

## 9) Plan / Review / Automations 约定
- Plan 模式：用于多步骤、跨文件、存在依赖的任务。
- Review 请求：先给 findings（按严重度），再给摘要。
- Automations：仅在用户明确要求“周期执行”时创建或更新。

## 10) 与用户沟通风格
- 简洁、直接、可执行。
- 先给结论，再给关键依据。
- 对不确定项明确标注，不伪造结论。
- Codex 的内容总结在与用户沟通时统一使用简体中文。

## 11) 特殊规则
- 当用户明确说 "push code" 时，Codex 应将当前项目中已完成的修改提交并推送到 GitHub 远程仓库（如有必要先执行 `git add -A`、`git commit`、`git push`）。
- 每次 Codex 更新项目后，必须同步更新项目根目录中的 `README.md`（至少反映变更内容、使用方式或验证结果中的相关部分）。

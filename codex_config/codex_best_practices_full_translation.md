# Best Practices

**如何开始使用 Codex，并通过经过验证的实践方法获得更好的效果。**

如果你刚开始使用 Codex，或者刚开始使用 **coding
agents（代码代理）**，本指南可以帮助你更快获得好的结果。

本文总结了一些让 Codex 在以下环境中表现更好的核心习惯：

-   Codex CLI
-   IDE 扩展
-   Codex App

内容涵盖：

-   Prompt（提示词）编写
-   Planning（任务规划）
-   Validation（验证）
-   MCP 集成
-   Skills 构建
-   自动化工作流

Codex
在你把它当作一次性助手时也能工作，但当你把它当成一个**可以持续配置和优化的开发伙伴**时，它的效果会更好。

一个推荐的整体思路是：

1.  提供正确的任务上下文
2.  使用 `AGENTS.md` 保存长期指导规则
3.  将 Codex 配置成适合你的开发流程
4.  使用 MCP 连接外部系统
5.  将重复任务沉淀为 Skills
6.  将稳定流程自动化

------------------------------------------------------------------------

# 强化第一次使用：上下文与提示

即使你的提示并不完美，Codex 仍然通常可以工作。

在很多情况下，你几乎不做任何准备，只是把一个复杂问题交给
Codex，它仍然可以生成有用的结果。

不过：

> 更清晰的提示可以显著提升可靠性，尤其是在大型代码库或高风险任务中。

如果你在 **大型或复杂仓库** 中工作，最重要的改进方式是：

**为 Codex 提供正确的上下文，以及清晰的任务结构。**

一个推荐的提示结构包括四个部分：

## Goal（目标）

你希望 Codex 构建或修改什么？

例如：

-   修复 bug
-   实现新功能
-   重构模块
-   优化性能

------------------------------------------------------------------------

## Context（上下文）

哪些信息与任务相关，例如：

-   代码文件
-   目录
-   文档
-   示例代码
-   报错日志

你可以使用 `@filename` 的方式将文件加入上下文。

------------------------------------------------------------------------

## Constraints（约束）

Codex 需要遵循哪些规则，例如：

-   编码规范
-   架构限制
-   安全策略
-   项目约定

------------------------------------------------------------------------

## Done when（完成条件）

什么情况下任务算完成，例如：

-   测试全部通过
-   Bug 不再复现
-   行为符合预期

这样可以帮助 Codex：

-   明确任务边界
-   减少错误假设
-   生成更容易 review 的代码

------------------------------------------------------------------------

# 推理等级（Reasoning Level）

Codex 支持不同的推理强度。

根据任务复杂度选择不同等级：

### Low

适合：

-   快速任务
-   范围明确的修改

### Medium / High

适合：

-   复杂逻辑修改
-   调试问题

### Extra High

适合：

-   长时间任务
-   多步骤复杂推理
-   复杂代码代理工作

------------------------------------------------------------------------

# 复杂任务先规划

如果任务：

-   很复杂
-   描述不清晰
-   有很多不确定性

建议 **先规划再写代码**。

------------------------------------------------------------------------

# 使用 Plan Mode

命令：

    /plan

或快捷键：

    Shift + Tab

Plan Mode 会：

1.  收集上下文
2.  提出澄清问题
3.  制定实现计划
4.  然后开始写代码

------------------------------------------------------------------------

# 使用 AGENTS.md

`AGENTS.md` 可以理解为：

> 一个专门给 AI agent 使用的 README。

它通常包含：

-   项目结构
-   构建命令
-   测试命令
-   lint / format
-   PR 规范
-   完成标准

------------------------------------------------------------------------

# 创建 AGENTS.md

CLI 中可以使用：

    /init

生成模板。

------------------------------------------------------------------------

# AGENTS.md 层级

    ~/.codex/AGENTS.md
    repo/AGENTS.md
    repo/module/AGENTS.md

离代码越近优先级越高。

------------------------------------------------------------------------

# 配置 Codex

个人配置：

    ~/.codex/config.toml

仓库配置：

    repo/.codex/config.toml

------------------------------------------------------------------------

# Sandbox 与权限

Codex 提供 sandbox 机制。

两个关键设置：

### Approval Mode

控制执行命令是否需要用户批准。

### Sandbox Mode

控制文件访问权限。

建议默认使用严格权限。

------------------------------------------------------------------------

# 提高可靠性

推荐工作流：

1.  写代码
2.  写测试
3.  运行测试
4.  lint / type check
5.  review diff

------------------------------------------------------------------------

# /review

命令：

    /review

用于：

-   PR Review
-   commit review
-   未提交代码检查

------------------------------------------------------------------------

# 总结

Codex 高效使用的关键：

1.  提供清晰上下文
2.  复杂任务先规划
3.  使用 AGENTS.md 固化规则
4.  使用 config 保证一致性
5.  使用测试和 review 提高可靠性

核心理念：

> 把 Codex 当作长期协作的开发伙伴，而不是一次性工具。

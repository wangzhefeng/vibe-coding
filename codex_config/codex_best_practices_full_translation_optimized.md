# Best Practices

**如何开始使用 Codex，并通过经过验证的实践方法获得更好的效果。**

如果你刚开始使用 **Codex**，或者刚开始使用 **代码代理（coding agents）**，这份指南可以帮助你更快获得良好的结果。

本文总结了一些能够让 Codex 在不同环境中表现更好的核心使用习惯，包括：

- Codex CLI
- IDE 扩展
- Codex App

内容涵盖：

- Prompt（提示词）编写
- Planning（任务规划）
- Validation（验证）
- MCP 集成
- Skills 构建
- 自动化工作流

Codex 在你把它当作 **一次性助手** 时也能正常工作。但当你把它当作一个 **可以持续配置和改进的开发伙伴** 时，它的效果通常会更好。

一个推荐的整体工作思路是：

1. 提供正确的任务上下文
2. 使用 `AGENTS.md` 保存长期指导规则
3. 将 Codex 配置为适合你的开发流程
4. 使用 MCP 连接外部系统
5. 将重复任务沉淀为 Skills
6. 将稳定流程自动化

------------------------------------------------------------------------

## 强化首次使用：上下文与提示（Context and Prompts）

即使你的提示并不完美，Codex 通常仍然能够工作。

在很多情况下，你几乎不需要做额外准备，只需要把一个复杂问题直接交给
Codex，它就能生成不错的结果。

不过需要注意的是：

> 更清晰的提示可以显著提升结果的可靠性，尤其是在大型代码库或高风险任务中。

如果你在 **大型或复杂仓库** 中工作，最重要的优化方式是：**为 Codex 提供正确的上下文，以及清晰的任务结构。**

推荐在 Prompt 中包含以下四个部分：

------------------------------------------------------------------------

### 1️⃣ Goal（目标）

你希望 Codex **构建或修改什么**？

例如：

- 修复某个 Bug
- 实现新功能
- 重构某个模块
- 优化性能问题

------------------------------------------------------------------------

### 2️⃣ Context（上下文）

哪些信息与任务相关，例如：

- 代码文件
- 目录结构
- 项目文档
- 示例代码
- 报错日志

在 Codex 中可以使用：

    @filename

将文件加入上下文。

------------------------------------------------------------------------

### 3️⃣ Constraints（约束）

Codex 需要遵循哪些规则，例如：

- 编码规范
- 架构限制
- 安全策略
- 项目约定

------------------------------------------------------------------------

### 4️⃣ Done When（完成条件）

什么情况下任务算完成，例如：

- 测试全部通过
- Bug 不再复现
- 功能行为符合需求

这种结构可以帮助 Codex：

- 明确任务边界
- 减少错误假设
- 生成更容易审查（review）的代码

------------------------------------------------------------------------

### 推理等级（Reasoning Level）

Codex 支持不同的推理强度，可以根据任务复杂度选择。

  推理等级     适用场景

  ------------ ------------------------

  Low          快速、范围明确的任务
  Medium       一般复杂度开发任务
  High         调试或复杂逻辑修改
  Extra High   长时间、多步骤复杂任务

不同团队的最佳配置可能不同，因此建议根据实际情况进行测试。

------------------------------------------------------------------------

### 提供上下文的技巧

在 **Codex App** 中可以使用 **语音输入**来描述需求。

很多开发者发现：

> 用语音描述复杂需求通常比手动输入更快。

------------------------------------------------------------------------

## 复杂任务先规划（Plan First）

如果任务：

- 很复杂
- 描述不够清晰
- 存在很多不确定性

建议 **先规划，再写代码**。

------------------------------------------------------------------------

### 使用 Plan Mode

最简单的方法是使用 **Plan Mode**。

命令：

    /plan

或快捷键：

    Shift + Tab

Plan Mode 会：

1. 收集上下文
2. 提出澄清问题
3. 制定实现计划
4. 然后开始写代码

这种方式可以显著提高复杂任务的成功率。

------------------------------------------------------------------------

### 让 Codex 先采访你

如果你的需求还比较模糊，可以让 Codex **先向你提问**。

例如：

> 在写代码之前先问我问题，以帮助明确需求。

Codex 可以：

- 挑战你的假设
- 发现需求漏洞
- 帮助你明确目标

然后再进行实现。

------------------------------------------------------------------------

### 使用 PLANS.md

对于更复杂的开发流程，可以使用：

    PLANS.md

这种方式适用于：

- 长周期开发任务
- 多步骤实现
- 多 Agent 协作

------------------------------------------------------------------------

## 使用 AGENTS.md 复用指导信息

当某些 Prompt 反复有效时，不要每次都重新输入。

可以将这些规则写入：

    AGENTS.md

你可以把它理解为：

> 一个专门为 AI Agent 编写的 README。

Codex 会自动读取这个文件。

------------------------------------------------------------------------

### AGENTS.md 应包含内容

一个好的 `AGENTS.md` 通常包括：

- 仓库结构说明
- 关键目录说明
- 如何运行项目
- 构建命令
- 测试命令
- lint / format 命令
- 工程规范
- PR 要求
- 禁止行为
- 完成标准

------------------------------------------------------------------------

### 创建 AGENTS.md

CLI 中可以使用：

    /init

自动生成模板，然后根据项目情况进行修改。

------------------------------------------------------------------------

### AGENTS.md 层级结构

项目中可以存在多个 `AGENTS.md`：

    ~/.codex/AGENTS.md
    repo/AGENTS.md
    repo/module/AGENTS.md

规则：

> 离当前代码目录越近的文件优先级越高。

------------------------------------------------------------------------

## 使用配置保证一致性

配置可以让 Codex 在不同会话中保持一致行为。

可以配置：

- 默认模型
- 推理等级
- sandbox 模式
- approval 模式
- profiles
- MCP 服务器

------------------------------------------------------------------------

### 推荐配置结构

个人配置：

    ~/.codex/config.toml

仓库配置：

    repo/.codex/config.toml

临时配置：

CLI 参数

------------------------------------------------------------------------

### config.toml 用途

该文件用于保存长期偏好，例如：

- MCP servers
- profiles
- agent 配置
- 实验性功能

可以：

- 手动编辑
- 或让 Codex 自动更新

------------------------------------------------------------------------

### Sandbox 与权限

Codex 提供 **操作系统级 Sandbox 机制**。

两个关键控制参数：

#### Approval Mode

控制：

- 什么时候需要用户批准
- 是否允许执行命令

#### Sandbox Mode

控制：

- 文件读写权限
- 目录访问范围

建议：

> 默认使用严格权限，仅在必要时放宽。

------------------------------------------------------------------------

### 常见配置问题

很多 Codex "效果不好"的问题其实来自配置问题，例如：

- 工作目录错误
- 文件权限不足
- 默认模型设置不合适
- 缺少必要工具
- 外部系统未连接

------------------------------------------------------------------------

## 使用测试与代码审查提高可靠性

不要只让 Codex 修改代码。

还应该让它：

1. 编写测试
2. 运行测试
3. 执行 lint
4. 检查类型
5. 验证行为
6. 审查 diff

------------------------------------------------------------------------

### Diff Review

在 Codex App 中可以打开 **Diff Panel**。

可以：

- 查看每一行代码修改
- 添加反馈
- 将反馈作为下一轮上下文

------------------------------------------------------------------------

### /review 命令

Codex 提供：

    /review

用于：

- PR Review
- 未提交代码检查
- Commit Review
- 自定义 Review

------------------------------------------------------------------------

### 团队代码审查规则

如果团队有：

    code_review.md

并在 `AGENTS.md` 中引用，

Codex 在执行 Review 时也会遵循这些规则。

------------------------------------------------------------------------

### Codex 在 PR 审查中的应用

在 GitHub Cloud 中，可以配置 Codex：

- 自动审查 PR
- 或在 `@Codex` 时触发审查

在 OpenAI 内部：

> Codex 会审查 **100% 的 Pull Requests**。

------------------------------------------------------------------------

# 总结

Codex 的高效使用通常遵循以下流程：

1️⃣ 提供清晰上下文
2️⃣ 复杂任务先规划
3️⃣ 使用 `AGENTS.md` 固化规则
4️⃣ 使用 `config.toml` 保证一致性
5️⃣ 使用测试和 Review 提高可靠性

核心理念：

> 把 Codex 当作长期协作的开发伙伴，而不是一次性的 AI 工具。

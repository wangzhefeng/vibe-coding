# Claude Code 知识库

Claude Code 是 Anthropic 推出的代理式编程工具。它的核心价值不是“聊天写代码”，而是让模型在终端、IDE、仓库和 Git 工作流里更完整地参与真实开发任务。

## 1. 知识介绍

### 1.1 什么是 Claude Code

根据官方仓库说明，Claude Code 是一个运行在终端中的 agentic coding tool，能够理解代码库、执行日常开发任务、解释复杂代码，并处理 Git 工作流。

它可以在以下场景工作：

- 终端 CLI；
- IDE 集成；
- GitHub 协作链路。

### 1.2 适用场景

- 代码探索、重构、补测试；
- 跨文件改动；
- 带工具调用的排障；
- 需要结合终端命令、版本控制与项目规范的任务。

### 1.3 与普通 AI 编程助手的区别

Claude Code 更强调：

- 长任务执行；
- 真正使用 shell / git / 项目规则；
- 更强的上下文管理；
- 将“提问”升级为“执行任务”。

## 2. 知识原理

### 2.1 核心运行模型

Claude Code 的工作方式可以概括为：

1. 读取用户目标；
2. 读取代码库与约束文件；
3. 在终端/工具环境中探索事实；
4. 形成计划并逐步执行；
5. 运行验证；
6. 汇报结果。

它的能力不只是生成代码，还包括“借助环境拿到 ground truth”。

### 2.2 为什么 AGENTS.md / 项目规则重要

这一类代理式工具会遵守仓库内的人类指令文件，例如 `AGENTS.md`。这会直接影响：

- 哪些文件能改；
- 测试怎么跑；
- 输出格式是什么；
- 是否需要更新 README；
- 是否可以提交或推送。

因此 Claude Code 的效果，既取决于模型，也取决于项目规则设计。

### 2.3 多代理与技能

从最新公开能力与社区实践看，Claude Code 正在从“单代理 CLI”向更强的协作系统演进，常见能力包括：

- 子代理 / teammate / worker 模式；
- 技能与插件；
- 更强的代码智能，如 LSP 能力；
- IDE 与命令行的一体化工作流。

## 3. 知识实践

### 3.1 入门流程

官方仓库给出的最小流程是：

1. 安装 Claude Code；
2. 进入项目目录；
3. 运行 `claude`；
4. 用自然语言描述任务。

### 3.2 推荐使用方式

#### 场景 A：代码库探索

让 Claude Code 做：

- 架构梳理；
- 文件定位；
- 调用链分析；
- 风险点审查。

#### 场景 B：明确目标的小步改动

典型提示方式：

- 目标；
- 上下文；
- 约束；
- 验收标准。

这种方式比模糊地说“帮我优化一下”效果更稳定。

### 3.3 最佳实践

- 先让它读规则、读代码，再改；
- 把任务拆成可验证的小步；
- 明确要求测试、类型检查或手动验证路径；
- 对高风险操作加入审批或人工复核；
- 尽量让仓库内说明文件持续更新。

### 3.4 常见坑

- 把它当成纯聊天工具，不给执行边界；
- 让它在未读项目规则时直接大改；
- 不做验证，只看生成结果；
- 对长任务不给中间检查点。

## 5. 其他重要内容

### 5.1 微信公众号文章提炼

从你收录的 Claude Code 微信文章和公开视频整理后，可以把社区高频经验归纳成下面几条：

- `Claude Code 的核心不是聊天，而是任务闭环`：好的使用方式一定包含探索、约束、修改、验证和回报，而不是只让它“生成点代码”。
- `从简单任务起步，比一上来做大项目更重要`：很多实战作者都强调先做 MVP，再逐渐扩展，不然重构成本会很高。
- `终端与规则文件同样重要`：Claude Code 的表现和终端环境、仓库结构、AGENTS.md、记忆文件高度相关。
- `把常见动作固化成 Command / Skill / Hook，比不断重写 prompt 更有效`。

### 5.2 “从零搓一个 Claude Code”类文章透露的工程经验

这类文章通常不是在讲“怎么模仿 UI”，而是在拆解 Claude Code 这类编码代理为什么能工作，核心经验包括：

- 把不受控的大命令拆成更原子的工具；
- 尽量用结构化工具调用，降低自由文本协议的不稳定性；
- 做上下文治理，避免长对话中上下文腐烂；
- 加入 trace、快照、检查点，让问题可复现、可回滚；
- 在模型足够强时，用 task / subagent 代替过于僵硬的 todo 机制。

### 5.3 社区实战里常见的工作流

高频工作流通常是：

1. 让 Claude Code 先读项目与规则；
2. 让它输出任务理解和计划；
3. 先做最小可验证改动；
4. 跑测试或做手动验证；
5. 根据结果继续迭代，而不是一次性全改。

这和“先写一堆需求，再一把梭生成整个项目”的方式有本质区别。

### 5.4 对新手最有价值的经验

公众号文章里很多“隐藏命令”“42 个技巧”类内容，真正长期有用的其实不是命令本身，而是以下习惯：

- 会描述清楚目标、上下文、约束、验收；
- 会让代理先探索事实；
- 会要求它给出验证路径；
- 会把有效经验沉淀为项目规则与技能。

### 5.5 Claude Code 的真正门槛

很多人以为门槛在“命令怎么写”，其实更大的门槛在：

- 是否会定义清晰任务；
- 是否会设置规则与边界；
- 是否会做验证与回滚；
- 是否会把经验沉淀成可复用流程。

### 5.6 与 Skills、MCP、Agent 的关系

- 与 `Skills`：可把常见任务沉淀为可复用技能；
- 与 `MCP`：可通过标准协议接更多工具与数据；
- 与 `Agent`：Claude Code 是“开发场景中的代理产品化形态”。

# Skills 知识库

Skills 是近一年 Agent 工具链里非常关键的一个抽象：它把“如何完成某类任务的经验、工具和约束”封装成可复用单元，让模型在合适的时候自动或半自动调用。

## 1. 知识介绍

### 1.1 什么是 Skill

在 OpenClaw / Claude Code / AgentSkills 一类体系里，Skill 通常是一个目录，核心文件是 `SKILL.md`，其中包含：

- 技能名称与描述；
- 触发条件或适用场景；
- 操作说明；
- 可选脚本、资源、工具定义。

它本质上不是普通知识文档，而是“给 Agent 的可执行说明书”。

### 1.2 Skill 解决什么问题

Skill 主要解决三个问题：

- 让高频任务从“每次临时提示”变成“稳定套路”；
- 让团队经验沉淀为可复用资产；
- 让模型在更小上下文内拿到更具体的任务指令。

### 1.3 Skill 与 Prompt / Tool 的区别

| 概念 | 作用 |
| --- | --- |
| Prompt | 单次对话或任务的即时指令 |
| Tool | 可被模型调用的能力接口 |
| Skill | 把任务说明、工具使用方式、约束与资源组织在一起 |

因此 Skill 常常是 Prompt 与 Tool 之间的桥梁。

## 2. 知识原理

### 2.1 技能的基本工作机制

Skill 的核心机制不是“多一个文件”，而是“把上下文压缩成可复用能力单元”：

1. 系统扫描技能目录；
2. 根据环境、配置、依赖、权限决定某个 skill 是否可用；
3. 将可用 skill 的摘要注入系统提示或命令面板；
4. 用户显式调用，或模型在合适时选择它；
5. 按 `SKILL.md` 的说明调用脚本、工具、资源。

### 2.2 设计原则

好的 Skill 通常满足：

- 任务边界清晰；
- 输入输出明确；
- 与现有工具映射明确；
- 风险与失败条件写清楚；
- 尽量短、小、专。

差的 Skill 常见表现：

- 过于宽泛，像一段空泛 SOP；
- 同时覆盖太多目标；
- 只写“你应该很聪明地完成任务”，没有可执行细节；
- 没有说明何时不该触发。

### 2.3 技能加载与优先级

以 OpenClaw 为例，技能通常有多层来源：

- 内置 bundled skills；
- 本地 managed/local skills；
- workspace skills；
- 插件提供的 skills。

加载优先级与环境过滤决定了“同名 skill 谁生效、当前能否被模型看见”。

## 3. 知识实践

### 3.1 最小 Skill 结构

```md
---
name: hello_world
description: 一个最小的示例 skill
---

# Hello World

当用户要求问候时，调用现有输出工具返回固定问候语。
```

如果需要脚本与资源，可在同目录继续放：

- `scripts/...`
- `templates/...`
- `assets/...`

### 3.2 设计 Skill 的推荐流程

1. 先找高频、重复、易错任务；
2. 把它拆成明确输入、过程、输出；
3. 决定是否需要脚本或模板；
4. 写短描述，让系统更容易正确触发；
5. 用真实样本反复测试命中率与执行效果。

### 3.3 适合做成 Skill 的场景

- 代码审查模板；
- 文档整理、报告生成；
- 多稿合并；
- 固定格式数据提取；
- 项目级初始化与常规检查；
- 调某个 MCP / API / 本地脚本完成的固定动作。

### 3.4 最佳实践

- 技能名称短、语义强；
- 描述写“何时用”，不是写“它很强大”；
- 指令写“怎么做 + 何时停止 + 失败怎么办”；
- 与工具配合时给出明确参数约束；
- 定期用真实任务回放评估 skill 的触发准确率与结果质量。

### 3.5 常见坑

- 为了“看起来完整”写成超长技能，反而降低触发质量；
- 把多个不相干任务塞进同一个 skill；
- 技能描述太弱，系统无法正确选择；
- 缺少版本管理、评估和淘汰机制。

## 5. 其他重要内容

### 5.1 微信公众号文章提炼

结合你收录的 Skills 相关文章与可访问的公开转译内容，社区对 Skills 的共识大致有 5 点：

- `Skill 是经验封装，不是知识收藏`：重点不在“告诉模型一些事实”，而在“告诉模型什么时候、按什么步骤、借助什么工具去做事”。
- `触发描述比正文更重要`：很多失败的 skill 不是内容不够长，而是名字和描述不够清晰，导致系统根本不会在对的时机选它。
- `Skill 应该与模型能力匹配`：工具越多并不一定越好，应该让 skill 贴合模型当前能稳定完成的操作。
- `渐进式披露比一次塞完更有效`：主 `SKILL.md` 只放最必要说明，进阶细节放引用文件、脚本或模板里，按需展开。
- `Skill 要能评估`：越来越多文章强调，不要“凭感觉写 skill”，而要用真实任务评估命中率、执行成功率和复用价值。

### 5.2 从 Claude Code 内部实践得到的启发

结合《构建 Claude Code 的经验：我们如何使用 Skills》相关公开转译，可以把值得保留的方法论总结为：

- 工具设计既是科学也是艺术，要观察模型真实输出再决定 skill 形态；
- 当模型能力提升后，旧的提示与旧的工具设计可能反过来限制它；
- 新功能不一定要通过新增工具实现，也可以通过 skill + 文档检索 + 子代理实现“渐进式披露”；
- skill 不只是执行模板，也可以承担“如何搜索更多上下文”的职责。

### 5.3 更实用的 Skill 编写模板

建议一个技能至少写清楚以下内容：

- 何时触发；
- 不该在什么情况下触发；
- 需要哪些输入；
- 期望输出是什么；
- 失败时如何回退；
- 是否依赖脚本、MCP、环境变量或外部 API。

如果这些问题没有写清，skill 往往会沦为“看起来很专业，但触发和执行都不稳定”的装饰文件。

### 5.4 为什么现在很多文章强调 skill-creator

你列出的多篇文章都在讨论 `skill-creator`，其背后反映的是技能工程进入了“可评估、可迭代”的阶段：

- 不再只追求把 skill 写出来；
- 开始关心命中率、误触发率、描述质量；
- 开始用批量样本测试 skill 的效果；
- 开始把技能看成产品资产，而不是一次性 prompt。

### 5.5 为什么 Skills 对 Agent 很重要

Agent 的瓶颈很多时候不是“模型不知道”，而是：

- 不知道何时该用什么套路；
- 不知道某个工具的正确使用边界；
- 每次都从零组织执行步骤。

Skill 的价值就是把这些经验显式化、可复用化、可治理化。

### 5.6 Skill 不是越多越好

技能过多会带来：

- 系统提示膨胀；
- 误触发概率上升；
- 管理和维护负担变重。

更推荐“少而精”的技能库。
